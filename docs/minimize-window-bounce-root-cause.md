# 浏览器最小化后弹回问题排查报告

## 1. 问题概述

在听书项目的前端页面中，用户点击浏览器最小化按钮后，窗口会立刻弹回前台，表现为“无法保持最小化状态”。

这个问题具有几个很迷惑人的特点：

- 在业务页面里几乎稳定复现
- 纯静态页面不复现
- 没有明显的 `window.focus()`、`alert()`、系统通知之类的显式抢焦点代码
- 最终根因不在业务代码，而在 `vue-router` 的浏览器 history 持久化逻辑

本报告记录完整排查过程、误判原因、最终根因和修复方案。

## 2. 复现环境

- 操作系统：Windows
- 运行方式：浏览器访问 WSL2 中启动的前端服务
- 前端技术栈：Vue 3 + Vue Router + Vite + Element Plus
- 访问场景：应用内浏览器 / 内嵌浏览器环境

## 3. 现象描述

### 3.1 表现

用户打开项目页面后，点击浏览器最小化按钮：

1. 窗口短暂进入失焦/隐藏状态
2. 很快又自动恢复为可见并重新获得焦点
3. 用户感知为“点了最小化，但浏览器又弹回来了”

### 3.2 影响

- 破坏基础窗口行为，用户无法正常切走应用
- 给人一种页面在“强行抢前台”的感觉
- 在桌面端使用体验上属于高优先级问题

## 4. 为什么这个问题一开始很容易判断错

这个问题之所以容易误判，不是因为现象少见，而是因为它的外观很像另外几类更常见的问题。

### 4.1 第一类误判：播放器或异步自动播放抢焦点

项目里确实存在几条“异步完成后自动播放”的链路：

- 试听完成后自动播放
- 单条合成完成后自动播放
- 长文本任务通过 SSE 完成后自动播放

这类行为在用户最小化窗口时非常可疑，因为浏览器或 WebView 在某些环境里会对媒体启动、权限、音频会话做前台处理。

因此最初把怀疑点放在播放器上是合理的，并不是拍脑袋。

### 4.2 第二类误判：输入框、对话框、下拉框的焦点管理

项目使用了 Element Plus，页面里有：

- 对话框
- 抽屉
- 下拉选择器
- 表单输入框

这类组件通常会做焦点管理、自动聚焦、关闭后回焦等动作。用户描述“最小化后又弹回来”，直觉上也很像“某个组件抢回焦点”。

### 4.3 第三类误判：问题出在某个具体业务页面

最开始问题是在真实业务页面里出现的，所以很容易沿着“工作台页面太复杂”“播放器在底部”“任务 SSE 回调有副作用”这些方向去挖。

但后来证明，这个 bug 的触发条件比业务页面更底层。

### 4.4 第四类误判：调试代码本身可能成为干扰项

排查期间曾经加过窗口生命周期调试逻辑，并在 `blur` / `visibilitychange` 时主动 `blur()` 当前元素来排除焦点问题。

这一步虽然是好意，但会带来一个风险：

- 调试代码自己可能在最小化瞬间修改焦点
- 从而成为新的干扰变量

这也是为什么后续必须把“主动 `blur()` 元素”的逻辑删除，只保留被动日志记录。

结论是：这个问题不是“看错了”，而是它一开始确实同时长得像播放器问题、焦点问题和页面问题。

## 5. 实际排查过程

### 5.1 第一步：先排播放器链路

先做了两类修正：

- 隐藏状态下不再自动播放异步返回的音频
- 对 `audio.play()` 的 Promise 拒绝做兜底

目的很直接：

- 如果是“最小化时异步开播导致窗口弹回”，这一步应该能立刻缓解
- 如果无效，就说明不是主要根因

结果：问题依旧存在。

这说明播放器可能是风险点，但不是这次最小化弹回的根因。

### 5.2 第二步：给窗口事件加观测

为了避免继续猜，后来给页面加了窗口生命周期日志，记录：

- `window-blur`
- `window-focus`
- `visibilitychange`
- 当前活动元素
- overlay 数量
- 是否正在播放

日志结果显示，每次点击最小化后，顺序大致都是：

1. `window-blur`
2. `visibility:hidden`
3. 很快又变成 `visibility:visible`
4. `window-focus`

同时：

- `isPlaying` 是 `false`
- 活动元素通常是 `BODY`
- 没有额外业务请求能解释“谁把窗口拉回来了”

这一步很关键，因为它第一次证明了：

- 弹回确实发生在浏览器窗口层
- 不是用户错觉
- 不是后端接口或音频播放直接触发

### 5.3 第三步：用纯静态页面做对照组

增加了一个完全不走 Vue 的静态探针页：

- 不加载 Vue
- 不加载 Router
- 不加载 Element Plus
- 只记录窗口事件

结果：静态页最小化完全正常，不会弹回。

这一步直接排除了“这个应用内浏览器天生就有最小化 bug”。

### 5.4 第四步：二分到 SPA 启动层

为了继续缩小范围，又做了几层对照：

- 空白 SPA 路由页
- 空白壳页（无播放器）
- 空白壳页（带播放器）
- 最小 Vue 页面
- 最小 Vue + Router 页面

结果是：

- 最小 Vue 页面正常
- 只要引入 Vue Router，就会弹回

这一步把问题从“大型业务页面”缩小到了“Router 层”。

## 6. 最终根因

### 6.1 直接根因

最终确认根因来自 `vue-router` 的浏览器 history 持久化逻辑。

在 `vue-router` 内部，history 实现会注册：

- `window.pagehide`
- `document.visibilitychange`

当页面进入隐藏状态时，它会执行类似下面的逻辑：

```js
if (document.visibilityState === 'hidden') {
  history.replaceState(
    { ...history.state, scroll: computeScrollPosition() },
    ''
  )
}
```

其目的本来是好的：在页面隐藏或离开时把滚动位置写回 `history.state`，以便后续导航恢复滚动状态。

### 6.2 为什么这会导致窗口弹回

在标准桌面浏览器里，这个行为通常没有问题。

但在当前环境中：

- 页面运行在应用内浏览器 / 嵌入式浏览器环境
- 浏览器窗口正处于“点击最小化后的过渡阶段”
- 这时 `vue-router` 在 `hidden` 阶段写入 `history.replaceState(...)`

这个 history state 写操作会触发当前容器环境的窗口激活行为，结果就是：

1. 页面刚进入 `hidden`
2. Router 立即写 `history.state`
3. 浏览器容器重新把窗口带回 `visible`
4. 用户看到“最小化刚点下去又弹回来”

### 6.3 为什么根因藏得这么深

因为项目代码里并没有写：

- `window.focus()`
- `alert()`
- 系统通知
- 明确的窗口控制逻辑

真正触发问题的是一个依赖内部为了“保存滚动位置”做的浏览器 state 写入。

这类问题很难直接从业务代码肉眼看出，所以必须依赖对照实验去逼近。

## 7. 最终修复方案

### 7.1 修复思路

不去改 `vue-router` 源码，也不去 fork 依赖，而是在创建 router history 的瞬间，临时拦截它注册的两个监听：

- `pagehide`
- `visibilitychange`

这样可以阻止 `vue-router` 在页面隐藏时执行那次 `history.replaceState(...)`。

### 7.2 实现位置

修复主要落在两个文件：

- `frontend/src/router-history.js`
- `frontend/src/router.js`

### 7.3 核心实现

`frontend/src/router-history.js` 提供了一个稳定版 history 创建器：

- 在创建 `createWebHistory()` 前
- 临时 patch `window.addEventListener` 和 `document.addEventListener`
- 忽略 `pagehide` 与 `visibilitychange` 这两类注册
- 创建完 history 后恢复原始事件注册函数

然后在 `frontend/src/router.js` 中：

- 不再直接使用 `createWebHistory()`
- 改用 `createStableWebHistory()`

## 8. 为什么这个修复有效

修复后的行为变化是：

- Router 仍然可以正常工作
- 路由切换、导航、页面渲染都不受影响
- 但页面最小化进入 `hidden` 时，不再触发那次 `history.replaceState(...)`

也就是说，真正触发窗口回弹的那一刀被拿掉了。

最终验证结果表明：

- 最小 Vue + Router 探针恢复正常
- 主项目页面恢复正常
- 用户可以稳定最小化浏览器窗口

## 9. 这次为什么会“犯这个错”

这里的“犯错”更准确地说，是排查顺序里的自然误导，而不是粗心。

### 9.1 现象和根因距离太远

用户看到的是“窗口弹回”，但根因其实是：

- 页面隐藏
- Router 写 history state
- 内嵌浏览器对该写操作做了异常窗口激活

这条链路太间接，第一眼不会有人优先怀疑 Router。

### 9.2 项目里确实存在很多更像元凶的东西

例如：

- 异步自动播放
- SSE 完成回调
- Element Plus 焦点管理
- 页面输入框和弹窗

这些怀疑都不是空穴来风，而是合理的优先排查对象。

### 9.3 依赖内部行为默认是“隐身”的

业务代码里看不到 `vue-router` 在 `hidden` 时会调用 `history.replaceState()`。

如果不去做：

- 对照页
- 最小探针
- 依赖源码搜查

几乎不可能第一时间想到这里。

### 9.4 调试时很容易把“相关性”误当成“因果性”

比如一开始发现问题出现在工作台页，很容易误判为：

- 是播放器导致的
- 是某个复杂页面导致的
- 是某个具体组件导致的

但实际上，工作台页只是“恰好走了 Router 的正常入口”，并不是根因。

## 10. 经验总结

### 10.1 排查窗口/焦点问题时，先做最小对照页

如果一开始就有：

- 纯静态页
- 最小 Vue 页
- 最小 Vue + Router 页

这个 bug 会更快收敛到 Router 层。

### 10.2 调试代码尽量只观察，不要干预

像主动 `blur()` 元素这种动作，虽然看起来像“帮助排除焦点问题”，但它本身就会改变系统行为。

更好的方式是：

- 先只打日志
- 确认时序
- 再做最小干预

### 10.3 对嵌入式浏览器环境要提高警惕

很多在普通浏览器里无害的行为，在 WebView、应用内浏览器、远程桌面或特殊窗口宿主里会变成窗口级 bug。

尤其要注意这些 API：

- `history.pushState` / `history.replaceState`
- 媒体自动播放
- 焦点管理
- 页面隐藏/恢复钩子

## 11. 副作用与取舍

这次修复的代价是：

- `vue-router` 在页面隐藏/离开时，不再自动把滚动位置写入 `history.state`

这意味着某些依赖该机制的滚动恢复场景会稍弱一点。

但和“窗口无法最小化”相比，这个代价是可接受的，而且目前没有看到对主流程造成明显损伤。

## 12. 最终结论

这次 bug 的真正根因不是业务页面，不是播放器，不是 Element Plus，也不是浏览器自己天生有问题。

真正的根因是：

- `vue-router` 在页面进入 `hidden` 时写 `history.replaceState(...)`
- 当前应用内浏览器环境会对这次写入做异常的窗口激活处理
- 导致最小化后的窗口立刻弹回

最终修复通过在 router history 创建时抑制这类持久化监听完成，问题已经被验证修复。
